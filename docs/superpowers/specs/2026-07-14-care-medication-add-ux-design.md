# 케어 탭 약 추가 UX 재설계 + 레이아웃 교정 기획

- 날짜: 2026-07-14
- 대상 커밋 기준: `8c97b68` (feat(design): refine mobile design system)
- 상태: 사용자 검토 대기

## 1. 문제 정의

케어 탭(`CareModeScreen`)의 현재 구성은 위에서부터:

1. `CareSetupPanel` — "케어 플랜 기본값" 전체 폼 (항상 펼쳐짐)
2. "오늘 투약 확인" 아젠다 목록
3. "임시 투약 추가" 버튼 → `QuickMedicationForm`("빠른 투약 기록") 토글
4. "약 일정 관리" 요약 카드 (스케줄 최대 3개)
5. 컨디션 안내 카드 / 리포트 생성 버튼

핵심 문제 4가지:

- **P1 — 폼 중복 노출.** `CareSetupPanel`은 프로필의 `ProfileCareDefaultsPanel`("케어 기본값")과 필드가 사실상 동일한 대형 폼인데 케어 탭 최상단에 항상 펼쳐져 있다. 사용자 인상: "프로필에 있는 폼이 왜 또 있지?" 케어 탭의 첫 화면이 '오늘 할 일 확인'이 아니라 '설정 입력'이 된다.
- **P2 — 약 일정 3중 표시.** 같은 스케줄이 한 화면에 세 번 나타난다: ① CareSetupPanel 하단 schedule rows("오늘 기록으로 불러오기") ② 오늘 투약 확인 아젠다 ③ "약 일정 관리" 카드.
- **P3 — 단기 복용약 개념 부재.** 현재 선택지는 (a) 프로필 장기 루틴 등록, (b) 오늘 1회 임시 기록뿐. "3일치 항생제" 같은 단기 처방이 어디에도 맞지 않아, 사용자가 케어 탭의 전체 폼(P1)을 쓰게 되는 근본 원인.
- **P4 — 용어 불일치.** 진입 버튼 "임시 투약 추가" ↔ 폼 제목 "빠른 투약 기록" ↔ 아젠다 소스 라벨 "임시 투약". 같은 기능에 세 가지 이름.

## 2. 사용 시나리오와 정신 모델

| 시나리오 | 빈도 | 올바른 처리 위치 |
| --- | --- | --- |
| A. 오늘 스케줄된 약을 먹였는지 체크 | 매일 | 케어 탭 아젠다 (현행 유지) |
| B. 오늘만 먹인 약을 즉석 기록 | 가끔 | 케어 탭 "약 추가 → 오늘만" |
| C. 병원에서 받은 단기 처방(3~14일) 등록 | 가끔 | 케어 탭 "약 추가 → 며칠간" (신규) |
| D. 매일 반복되는 장기 루틴 관리 | 드묾 | 프로필 "케어 기본값" (유일한 관리 위치) |

정신 모델을 이렇게 고정한다: **프로필 = 장기 루틴 원장(ledger), 케어 탭 = 오늘의 실행 + 단기 예외.** 폼이 두 곳에 있는 이유를 없애는 게 아니라, 두 곳의 폼이 서로 다른 질문에 답하게 만든다.

## 3. 접근안 비교

### A안 — 진입점 통합 + 기간별 분기 (추천)

케어 탭에서 `CareSetupPanel` 상시 노출을 제거하고, 단일 "**+ 약 추가**" 버튼이 기간 선택으로 분기하는 인라인 카드를 연다:

- **오늘만** → 기존 `QuickMedicationForm` 필드 (약 이름, 용량, 상태, 메모) — 즉석 1회 기록
- **며칠간** → 약 이름, 용량, 복용 시간, 시작일(기본 오늘)/종료일(필수) — 기존 `onSaveCareSetup` 파이프라인 재사용, 종료일이 있는 단기 스케줄 생성
- **매일 (장기)** → 폼 대신 안내문 + "프로필의 케어 기본값에서 관리하기" 이동 링크

장점: 폼 중복 해소, 시나리오 B·C·D가 각자 최소 필드만 봄, 기존 데이터 모델(스케줄 + endsOn) 그대로 재사용. 단점: 가드 스크립트(`verify-presentation-state.mjs`)의 `CareSetupPanel` 포함 강제 조항 갱신 필요, 구현 범위 중간.

### B안 — 최소 변경: CareSetupPanel 접기

폼은 그대로 두되 기본 접힘(collapsed) 상태로 두고 헤더 탭 시 펼침. 장점: 변경 최소, 가드 무수정. 단점: 중복 폼 자체는 남아 "왜 또 있지"가 "펼치면 또 있지"로 바뀔 뿐, 단기 처방(P3) 미해결.

### C안 — 완전 분리: 케어 탭은 기록 전용

스케줄 생성 UI를 케어 탭에서 전부 제거하고 단기 처방도 프로필에서 등록. 장점: 역할 구분이 가장 단순. 단점: "병원 다녀와서 케어 탭에서 바로 3일치 약 등록" 흐름이 프로필로 강제 이동되는 마찰, 가드의 의도("케어 흐름에서 플랜 생성 가능")와 정면 충돌.

**추천: A안.** 중복의 원인(장기 루틴 폼)만 프로필로 위임하고, 케어 탭에서 즉시 처리해야 자연스러운 것(오늘/단기)은 목적에 맞는 축소 폼으로 남긴다.

## 4. A안 상세 설계

### 4.1 케어 탭 화면 구성 (위 → 아래)

1. 세그먼트 (케어 / 리포트) — 현행 유지
2. **오늘 투약 확인** — 최상단으로 승격. 케어 탭의 첫 화면 = 오늘 할 일
3. **+ 약 추가** (SecondaryButton) — 누르면 아래에 인라인 SurfaceCard 확장:
   - 카드 최상단 SegmentedControl: `오늘만 | 며칠간 | 매일`
   - 기본 선택 "오늘만" (가장 빈번한 즉석 기록)
   - 선택에 따라 하위 필드 교체 (4.2)
4. **약 일정** 카드 — 저장된 스케줄의 **유일한** 요약 위치 (P2 해소)
   - 각 행: 시간 · 약 이름 · 용량, 우측 "오늘 기록으로 불러오기" (CareSetupPanel에서 이전)
   - 단기 스케줄에는 기간 뱃지: `~7/17까지`
   - 3개 초과 시 "외 N개 더 보기" 확장 (무언 truncation 제거)
   - 카드 하단 텍스트 링크: "매일 반복 루틴은 프로필에서 관리해요 →"
5. 컨디션 카드 / 리포트 생성 버튼 — 현행 유지

### 4.2 약 추가 카드 분기

| 분기 | 필드 | 저장 경로 |
| --- | --- | --- |
| 오늘만 | 병명(선택), 약 이름(필수), 용량, 오늘 투약량, 상태 세그먼트, 메모 | `onAddDose` (기존 QuickMedicationForm 로직 그대로) |
| 며칠간 | 병명(선택), 약 이름(필수), 용량, 복용 시간(TimePicker, 기본 1개 + 추가 가능), 시작일(기본 오늘), 종료일(**필수**, 기본 +7일) | `onSaveCareSetup` (기존 케어 플랜 파이프라인, endsOn 필수라는 점만 다름) |
| 매일 | 필드 없음. 안내문 + "프로필에서 관리하기" 버튼 | `PawBloomShell`에 탭 전환 콜백 추가 (`setActiveTab("settings")` + `setShowPetSettings(true)` 경로 재사용) |

- 저장 성공 시 카드 접힘 + 기존 `SaveFeedbackBar` 피드백, "며칠간" 저장 시 오늘 날짜가 기간에 포함되면 아젠다에 즉시 행이 생기므로 별도 확인 UI 불필요.
- 알림: "며칠간" 저장도 `saveCareSetupAndRefreshReminders`를 타므로 투약 알림이 자동 갱신됨 (기존 동작 재사용).

### 4.3 라벨/카피 (P4 해소)

- 진입 버튼: `+ 약 추가` (신규 키 `care.addMedication`)
- 분기 라벨: `오늘만` / `며칠간` / `매일` (신규 키 3개)
- "오늘만" 폼 제목: "오늘 투약 기록" — "빠른/임시" 용어 폐기
- 아젠다 소스 라벨: `일정 투약` 유지, `임시 투약` → `오늘만` 기록과 용어 통일 (예: "오늘 추가한 약")
- 미사용 키 `care.temporaryCopy`는 "오늘만" 분기 도움말로 활용하거나 삭제
- 프로필 쪽은 현행 유지 (`pet.careDefaultsSavedHint` "오늘 실제 투약 여부는 Care에서 체크해요"가 이미 역할 구분을 설명)

### 4.4 편집 흐름

- 아젠다 행의 편집은 현행처럼 화면 하단 폼으로 점프하지 않고, **해당 행 바로 아래 인라인 확장**으로 편집 폼을 연다 (맥락 유지).
- 행 우측 편집 트리거는 텍스트 링크 "투약 기록 수정"(저장 버튼 라벨 재사용 중) 대신 연필 아이콘 또는 "수정" 단축 라벨 + 44px 터치 타깃.

## 6. 제약 및 가드 스크립트 영향

`scripts/verify-presentation-state.mjs`와의 관계 — **반드시 함께 갱신해야 하는 항목**:

1. `careModeScreen.includes("CareSetupPanel")` 강제 조항: 가드의 의도는 "케어 흐름에서 플랜 생성이 가능해야 한다"이다. A안의 "며칠간" 폼이 `onSaveCareSetup`을 호출하므로 의도는 충족된다. 가드 문자열을 새 컴포넌트명(예: `CareMedicationAddCard`)으로 교체하고 주석으로 의도를 남긴다.
2. 세로 스택 가드 3건(diary meal, quick medication `inputGrid`, care setup `row`)은 **유지**. 이번 기획의 어떤 교정도 입력 필드를 row로 되돌리지 않는다. 교정 #1(actionButtons row)은 입력 필드가 아닌 고정 라벨 버튼이라 가드 대상 밖.
3. `docs/design/DESIGN_QA.md`는 `edit_policy: exclusive`(claude-opus-4.8-extra 소유)이므로 직접 수정하지 않는다. 가드 스크립트 변경 시 해당 문서가 참조하는 줄 번호(64-75)가 어긋날 수 있음을 커밋 메시지에 명시해 소유 모델이 후속 반영하게 한다.

## 7. 구현 단계 (개요)

1. **Step 1 — 레이아웃 교정 (기획 독립적):** 교정 #1, #2, #3, #6, #7 — CareModeScreen/패널 소규모 수정, 스냅샷 없는 스타일 변경 위주.
2. **Step 2 — 약 추가 카드:** `CareMedicationAddCard` 신설(medication context ui 레이어), 오늘만/며칠간/매일 분기, 신규 i18n 키, CareSetupPanel 제거, 가드 갱신. DDD 규칙상 케어 플랜 저장 호출은 props로 주입(현행 `onSaveCareSetup` 패턴 유지).
3. **Step 3 — 약 일정 카드 강화:** "오늘 기록으로 불러오기" 이전, 기간 뱃지, 더 보기, 프로필 이동 링크(+Shell 콜백).
4. **Step 4 — 정리:** 교정 #4, #5, #8, #9, #10, 미사용 키 정리, `npm run verify:presentation` + 기존 테스트 통과 확인.
